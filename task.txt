## Phase 1: Critical Fixes for generate-render (Urgent)

- [x] 1. Enable JWT verification on generate-render edge function
  - Remove verify_jwt = false for generate-render in supabase/config.toml
  - Add Authorization header validation at the top of the serve() handler
  - Return 401 for unauthenticated requests
  - This function calls a paid API (Gemini) — unauthenticated access is a financial risk

- [x] 2. Fix prompt hardcoded to "interior design" for all project types
  - buildDesignPrompt() in generate-render/index.ts always says "interior design render"
  - Add discipline-aware prompt mapping (interior, architecture, landscape, industrial)
  - Accept project_type in the request body from designerAgent.ts
  - Use discipline map: residential→"interior design render", architecture→"architectural exterior render", landscape→"landscape design visualization", etc.
  - Fix fallback description from "A well-designed interior space" to be discipline-aware

- [x] 3. Fix storage path mismatch with delete policy
  - Edge function stores renders at {projectId}/render-{timestamp}-{index}.png
  - Storage delete policy checks (storage.foldername(name))[1] = auth.uid()
  - Project UUIDs ≠ user UUIDs, so deletions silently fail
  - Add migration to fix the policy: check project ownership via projects table instead
  - Also affects user-uploaded files from EmptyBriefPrompt.tsx

- [x] 4. Fix imageConfig.aspectRatio nesting in Gemini API call
  - Current: aspectRatio nested under imageConfig within generationConfig — may be silently ignored
  - Correct: aspectRatio directly within generationConfig per Gemini API docs
  - Fix hardcoded resolution metadata ("1408x768") to reflect actual API response dimensions

- [x] 5. Add input validation on generate-render function
  - Validate project_id is a non-empty string
  - Validate description is under 2,000 characters
  - Validate style is one of the known values
  - Return 400 for invalid inputs
  - Sanitize description before injecting into prompt (prompt injection risk)

- [x] 6. Fix sequential generation timeout risk
  - Three sequential Gemini calls take 30-90 seconds, risking Supabase free tier 150s timeout
  - Option A: Generate one image per function call, call three times from client
  - Option B: Reduce to 1-2 variations and parallelize where possible
  - Add per-call timeout handling so partial results are returned rather than total failure
